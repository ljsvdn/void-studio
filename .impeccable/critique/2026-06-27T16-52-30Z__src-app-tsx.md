## Anti-Patterns Verdict

This does not immediately look AI-generated. The masthead grid, large display type, living shader, work-row treatment, and manifesto typography feel designed, not templated.

The risk is not generic SaaS slop. The risk is concept-site thinness: the page has a beautiful shell, but the work section does not yet prove enough. A skeptical creative director will likely admire the opening and then ask, "Where is the evidence?"

Detector status: deterministic detector failed with `Error: bundled detector not found.` Browser and source evidence were used instead. Production build strips the shader tuner via `import.meta.env.DEV`, so the visible dev tuner is not counted as a production issue.

## Overall Impression

The site has the right bones and a real point of view. I would not change the core direction: dark gallery, oversized grid masthead, living shader, disciplined typography, and one direct email CTA are correct for VØID.

I would change how much proof the page gives, how mobile users experience work, and a few technical semantics that currently weaken accessibility and polish.

## What's Working

1. The first viewport has presence. The page feels like a studio with taste rather than a converted SaaS template.
2. The process/manifesto type system is the strongest section. Outline/fill contrast plus italic pivot words creates a recognizable voice.
3. The implementation has good hardening already: reduced-motion handling, skip link, scroll restoration, shader pausing, strong contrast, semantic landmarks, and a passing TypeScript check.

## Priority Issues

**[P1] Selected Work does not prove enough**

Why it matters: The site exists to convert skeptical, design-fluent prospects. Three fictional-feeling project names plus generic categories are not enough to prove craft, depth, or taste.

Fix: Add inline project proof. Each row should expand or route to a real case-study surface with: one artifact, role/scope, constraint, outcome, and a small visual/material cue. Avoid modal-first design; inline expansion fits the editorial grid better.

Suggested command: `impeccable craft selected-work proof interaction`

**[P1] Mobile has real horizontal overflow from project shader layers**

Why it matters: Browser evidence found `documentElement.scrollWidth` wider than client width on 375px and 320px viewports. `body { overflow-x: hidden }` masks it visually, but it is still layout debt and can cause weird scroll/focus behavior.

Evidence: At 375px, document width was 427 vs 360 client width. At 320px, document width was 361 vs 305. Source is `.bodylayer { left: 78%; width: min(440px, 84vw) }`.

Fix: Contain the mobile bodylayer inside `.blendstage`, use transform-based positioning with max inline size, or clip the stage rather than relying on body-level overflow hiding.

Suggested command: `impeccable adapt selected-work mobile`

**[P1] Screen-reader heading text is concatenated**

Why it matters: Visual line breaks currently expose names like `We buildthe void`, `SelectedWork`, and `Let’s buildfrom nothing.` This is a small visual trick creating a real assistive-tech defect.

Fix: Add accessible labels or actual spacing around split heading parts. For example, `aria-label="We build the void"` on the heading and `aria-hidden` on decorative split spans.

Suggested command: `impeccable harden heading semantics`

**[P1] Work rows behave like links but go nowhere**

Why it matters: All project rows are anchors pointing to `#work`, inside the same section. Keyboard and screen-reader users encounter real links that do not take them to meaningful content. Sighted users also get a dead-end click.

Fix: Either make them buttons that select/reveal a project preview, or give them real hrefs. Given the current page, the stronger move is inline expandable proof panels.

Suggested command: `impeccable craft selected-work proof interaction`

**[P2] Touch/mobile work interaction is weaker than desktop**

Why it matters: Desktop gets a hover-driven luminous moment. Mobile mostly gets a list, so the most distinctive interaction is unavailable to a large part of the audience.

Fix: Tap-to-select a project row, pin its shader state, and reveal a compact proof panel below the row. The same interaction can replace the current no-op anchors.

Suggested command: `impeccable adapt selected-work mobile`

**[P2] Footer/social touch target for `X` is too small**

Why it matters: The one-character `X` link measured about 9x33px. It is legal text, but not a comfortable target.

Fix: Give social links a minimum inline hit area, or label it `X / Twitter` visually while keeping the restrained column design.

Suggested command: `impeccable harden footer links`

**[P2] WebGL fallback is not source-visible**

Why it matters: The page depends on five shader canvases. Reduced motion and offscreen pause are good, but if WebGL creation fails, the brand's central visual proof can disappear.

Fix: Add a static CSS gradient/fallback surface or error boundary around `PaperShader`.

Suggested command: `impeccable harden shader fallback`

## Persona Red Flags

**Jordan, first-timer:** The primary CTA is clear, but project rows are ambiguous. Clicking a project gives no meaningful confirmation or next destination.

**Sam, accessibility-dependent:** Landmarks and contrast are good. The heading concatenation and hover-led work meaning are the main red flags. Scanline/grain plus outline text can also tax low-vision users.

**Casey, distracted mobile:** Mobile is readable and compact, but the distinctive work preview interaction disappears. The hidden horizontal overflow is also a mobile-layout smell.

**Skeptical creative director:** The opening earns attention, but the portfolio does not yet withstand scrutiny. The artifact proves you can create a mood; it does not yet prove you can ship client outcomes.

## Minor Observations

- `mailto:` as the only contact path is acceptable for a studio page, but a copy-email affordance would be more resilient.
- The footer is now correctly full-width; browser evidence did not find footer overflow on mobile.
- Static contrast is strong: ink on background is about 17.05:1, dim text about 5.81:1.
- Bundle size is fine for a shader-led brand page: JS ~225.72 kB, gzip ~83.12 kB.
- Custom cursor RAF is a small performance risk, but mitigated by pointer/reduced-motion checks.

## Questions to Consider

1. Should `Selected Work` become lightweight inline proof, or full case-study routes?
2. Do you want the project data to stay fictional/demo-like for now, or should we make it feel more like real client evidence even before full case studies exist?
3. Is the shader palette intentionally broader than the PRODUCT.md oxblood/white discipline, or should the product context be updated to match the current blue/crimson/amber direction?
4. Should the footer remain email-first, or should we add copy-to-clipboard as a quiet secondary affordance?
